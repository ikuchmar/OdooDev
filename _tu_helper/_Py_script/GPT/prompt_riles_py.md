# Руководство по оформлению Python-кода

Компактный набор правил, применимый ко всем Python-скриптам в проекте.
Цель — простой, понятный, и хорошо документированный код.

## Краткий контракт
- В каждом файле есть русскоязычное описание назначения, входов/выходов, зависимостей и примера запуска.
- Перед каждой процедурой — разделитель `# ==============================` и краткое назначение функции.
- Комментарии и docstring'и объясняют неочевидную логику и контекст.
- Логирование через `logging`; валидация входов; чёткая обработка ошибок.
- Понятность для программиста среднего уровня — без «магии», с прямолинейной логикой.

## 1. Документация
- В начале файла: русскоязычный блок-описание (что делает, как работает, ввод/вывод, зависимости, пример запуска).
- У каждой функции: краткий комментарий назначения перед ней и информативный docstring внутри.
- Комментарии — по делу: объясняют почему и как, а не «что» очевидно из кода.

## 2. Стиль и структура
- Добавлять разделитель `# ==============================` перед каждой процедурой/крупным блоком.
- Разбивать задачи на маленькие функции; избегать излишней абстракции и вложенности.
- Использовать аннотации типов, `pathlib.Path` для путей, явные константы вместо «магических чисел».

## 3. Именование
- Выбирать осмысленные, самодокументирующие имена переменных и функций.
- Избегать сокращений и двусмысленности; единый стиль по всему файлу/проекту.

## 4. Ошибки и логирование
- Не использовать `print` для служебных сообщений — только `logging` (debug/info/warning/error).
- Валидировать входные данные; выбрасывать осмысленные исключения при неверных параметрах.
- Для фатальных ошибок — ненулевой код выхода; для recoverable — информативные логи.

## 5. CLI и пользовательский опыт
- Использовать `argparse` с понятными `help`, разумными значениями по умолчанию.
- Где возможно — поддерживать режим `--dry-run` (показать, что будет сделано, без побочных эффектов).

## 6. Работа с файлами и внешними командами
- Применять `pathlib`; при текстовых файлах указывать `encoding='utf-8'`.
- Создавать каталоги с `exist_ok=True`; проверять наличие внешних бинарей заранее.
- Команды для внешних процессов собирать списком, логировать и проверять `returncode`.

## 7. Понятность для программиста среднего уровня
- Писать максимально простым и ясным способом: прямолинейная логика, короткие функции.
- Избегать «магии» и скрытых побочных эффектов; явно обозначать ключевые шаги.
- Добавлять краткие пояснения в местах потенциальной неоднозначности.
